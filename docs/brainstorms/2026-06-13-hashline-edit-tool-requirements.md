# Hashline Edit Tool for Claude Code — Requirements

## Summary

A Claude Code plugin (`claude-hashline`) that replaces `str_replace`-style editing with **hashline**: an MCP tool suite that tags every read/grep line as `line:hash|content` and edits by those anchors, with the built-in `Edit`/`Write`/`NotebookEdit` tools hard-blocked so the model is forced onto it. A synthetic benchmark built on `claude-p` measures hashline against the built-in editor across Claude tiers, scored on edit-failure rate and token cost.

---

## Problem Frame

The built-in editor makes the model reproduce the exact old text — every character of whitespace and indentation — to anchor an edit. Near-misses are rejected with "String to replace not found in file," and the model burns turns and context retrying. The observed daily cost is retry loops and friction, not silent corruption.

A recent benchmark showed a hash-anchored edit format cut edit failures and output tokens sharply, with the largest gains on the weakest models (Grok Code Fast 1: 6.7%→68.3%). But it tested only non-Claude models. Claude Code runs Claude exclusively, and Claude is heavily tuned on its own `str_replace` format — so whether hashline helps *here* is unverified. This work exists to both relieve the retry-loop pain in real sessions and produce a credible Claude-specific measurement.

---

## Key Decisions

- **Full hashline suite + hard block (Architecture A).** Build the complete read/grep/edit path and deny the built-in mutators, rather than injecting tags via hook (would double-render content and pollute the token signal) or repairing near-miss `str_replace` (never tests hashline). Only the faithful end-to-end path yields an honest measurement.
- **Anchor = line number + short content hash.** The line number disambiguates; the hash verifies the line is what the model thinks it is. A mismatch means the file changed since the read, so the edit is rejected before touching the file. Exact hash algorithm, length, and normalization are deferred to planning.
- **Two block mechanisms, one tool.** A PreToolUse hook enforces the block in daily sessions; the benchmark instead toggles arms with `claude-p --disallowedTools`, so no reinstall is needed between runs.
- **Control arm is the unmodified built-in editor.** The thing hashline must beat is the real default, so the control allows native `Edit`/`Write` and disallows the hashline tools.
- **Test across Claude tiers.** Hypothesis, mirroring the article: weaker Claude tiers (Haiku) gain more than stronger ones (Opus).
- **Token and retry-loop reduction count as success even with no pass-rate gain.** Because Claude is `str_replace`-tuned, the honest outcome may be a flat success rate with lower tokens — that still wins the daily-value half.
- **v1 is read + edit only; grep follows.** Ship `hashline_read` + `hashline_edit` first to validate the loop and get a benchmark number, then add `hashline_grep`. Until grep ships, the model falls back to built-in `Grep` (untagged) and can `hashline_read` before editing.

## Requirements

### Hashline tools

- R1. `hashline_read` returns each line tagged `line:hash|content`.
- R2. `hashline_grep` returns matches with the same `line:hash|content` tagging. (Deferred to a later increment — see Scope Boundaries; not in v1.)
- R3. `hashline_edit` applies edits by anchor: replace-line, replace-range (`start..end`), insert-after, and delete.
- R4. `hashline_edit` also covers creating a new file and full-content replacement, so it is a strict superset of the blocked `Write`.
- R5. An edit whose referenced anchor's current hash does not match is rejected without modifying the file.
- R6. Editing a file that has had no hashline read in the current session forces a read first, so the model never anchors on hashes it has not seen.

### Block and escape

- R7. When the plugin is enabled, a PreToolUse hook denies built-in `Edit`, `Write`, and `NotebookEdit`, returning a message that redirects to the hashline tools.
- R8. Built-in `Read`/`Grep` stay available for non-text cases (images, PDFs, notebooks), but the model is steered toward `hashline_read`/`hashline_grep` for text.
- R9. A one-flip escape hatch disables the block without uninstalling the plugin, so a hashline bug never leaves editing impossible.

### Benchmark

- R10. Fixtures are generated by mutating real source files with reversible mechanical bugs (operator swaps, boolean flips, off-by-one, removed guard clauses), each paired with a plain-English task description.
- R11. The harness runs each task headless in an isolated workspace via `claude-p`, with a temporary copy of the file.
- R12. Two arms run per task: **hashline** (block on, built-in `Edit`/`Write` disallowed) and **control** (hashline tools disallowed, built-in editor allowed).
- R13. Each arm runs across multiple Claude tiers.
- R14. Each run captures edit-failure/rejection count, output tokens, and pass/fail (the post-edit file diffed against the expected file after formatting).
- R15. The harness emits a comparison report broken down by model and arm.

## Acceptance Examples

- AE1. **Stale-read rejection.** **Covers R5.** **Given** the model read a file and holds anchor `12:a3`, **and** the file's line 12 changed since (its hash is now `7c`), **when** the model issues `replace 12:a3`, **then** the edit is rejected with a hash-mismatch message and the file is unchanged.
- AE2. **Edit before read.** **Covers R6.** **Given** a file the model has not read this session, **when** the model calls `hashline_edit` on it, **then** the edit is refused with a message instructing it to `hashline_read` the file first.

---

## Success Criteria

- **Daily value:** with the block on in real sessions, edits land first-try at a clearly higher rate than the built-in editor, retry loops drop, and the escape hatch is never needed to work around a *capability* gap (only to dodge an outright bug).
- **Proof:** the benchmark produces a credible per-tier comparison of hashline vs control on edit-failure rate and output tokens, reported honestly including a null or negative pass-rate result if that is what the data shows.

## Scope Boundaries

### Deferred for later

- `hashline_grep` (R2) — deferred to a later increment. v1 is `hashline_read` + `hashline_edit` only.

### Outside this product's identity

- **Non-Claude models.** Claude Code runs Claude only, so the article's cross-model gains are unreachable here and are not a goal.
- **Polished, widely-distributed plugin.** This targets personal daily use plus a credible proof, not a marketplace release.
- **Reformatting built-in `Read`/`Grep` output.** Not possible from a plugin — this constraint is what forced Architecture A.
- **Cursor-style merge model, fuzzy patch formats, and the hook-injected-tag (B) or forgiving-shim (C) approaches** as the build. C's baseline is reused only as the benchmark control.

---

## Dependencies / Assumptions

- **Headless runner:** `claude-p` (`--cwd`, `--disallowedTools`, `--model`, `--max-turns`, exit codes, transcript token parsing). `shannon` is an alternative; `claude -p` / the Agent SDK is the robustness fallback, since `claude-p`'s own README frames it as fragile and educational.
- **Claude Code plugin mechanics:** MCP servers add namespaced tools (`mcp__hashline__*`); PreToolUse hooks can deny a tool call. Both are documented extension points.
- **Assumption (to validate):** with the built-in mutators denied, the model reliably selects the hashline tools. If tool-selection friction is high — Claude is not tuned on these tool names — stronger steering (naming, descriptions, deny-message wording) may be needed.

## Outstanding Questions

### Deferred to planning

- Exact hash algorithm, length (2 vs 3 chars), normalization (raw bytes vs trimmed), and within-file collision handling.
- Fixture corpus: the React codebase (as in the article), this repo, or another JS/TS project.
- Task count and runs-per-arm (the article used 180 tasks × 3 runs).
